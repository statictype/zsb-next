# Component audit — edition `Hero` / `ThemeArtists` + `artists/`

**Scope:** `src/app/(site)/editions/[year]/_components/{Hero,ThemeArtists}.{tsx,recipe.ts}`,
`src/app/(site)/artists/page.tsx` (its `_components/` is empty). Grounded against
`src/design-system/{tokens.ts,preset.ts}`, `src/components/{PageHero,EditionTheme,
ArtistsTable,Figure,ui/SectionHeading}`, `src/components/Navigation/Navigation.recipe.ts`,
and both call sites (`editions/[year]/edition-content.tsx`, `artists/page.tsx`).

**Raw-value policy applied** (quoted from `src/design-system/preset.ts:15-28`): `strictTokens`
is **ON**; every value is a token or a deliberate `[bracket]`. Brackets are **debt to justify
or remove, not blessed art direction** — the rule is *"second occurrence → token"* and *"snap
to nearest token, accept a small visual delta"*; a raw literal survives only when the exact
number is load-bearing (alignment/overlap geometry, hairline px, viewport/%-centering math).

This pair is one of the more disciplined corners of the codebase — both recipes carry dense,
accurate rationale comments, and no structural, type, or behavioral issues turned up. Every
finding below is reuse or documentation debt: exact-match compositions the recipes re-typed by
hand, and a few ramps/brackets that break this file's own comment convention. **Structure &
composition** and **types** are clean for this scope (Pick-narrowed props, no `any`/`!`, no
slot sprawl, no reinvented primitives — both components already delegate to `SectionHeading`,
`EditionTheme`, `Figure`, `ArtistsTable` correctly) — not repeated as findings below.

## Detailed findings

### DS-1 — `body` slot duplicates the `prose` textStyle

**file:line:** `src/app/(site)/editions/[year]/_components/ThemeArtists.recipe.ts:40-52`
**axis:** ds-hygiene
**risk tier:** no-op — the values match exactly, this is a pure reuse
**evidence:** `ThemeArtists.recipe.ts`'s `& p` block declares `fontFamily: 'body', fontSize:
'base', lineHeight: 'body', color: 'body'` — the *exact* definition of `textStyles.prose`
(`src/design-system/tokens.ts:511-513`), already consumed by four other recipes:
`src/app/(site)/privacy/page.recipe.ts:19`, `src/app/(site)/visit/_components/VisitFaq.recipe.ts:16`,
`src/app/(site)/partners/page.recipe.ts:39`, `src/components/PillarGrid/PillarGrid.recipe.ts:37`
— well past the DS's own "second occurrence → token" trigger. The two extra declarations here,
`fontWeight: 'regular'` and `textAlign: 'left'`, are the browser defaults for a `<p>` and add
nothing. `textWrap: '[pretty]'` is also written bracketed even though the DS's own `lead` /
`leadLarge` textStyles write the identical value unbracketed (`textWrap: 'pretty'`,
`tokens.ts:497,508`) — `text-wrap` has no token category, so the brackets here are just noise.
**recommendation:** Replace the block with `textStyle: 'prose'` + `textWrap: 'pretty'` (unbracketed,
matching the `lead` convention); drop the redundant `fontWeight`/`textAlign`. Zero visual change.
**effort:** trivial

### DEAD-1 — `& p:last-child` guards a case that can't occur

**file:line:** `src/app/(site)/editions/[year]/_components/ThemeArtists.recipe.ts:43,52`
**axis:** dead-code
**risk tier:** no-op
**evidence:** The recipe styles `& p` and `& p:last-child { marginBottom: '0' }` as if `body`
could render several paragraphs. But `edition.themeSection` is typed `ThemeData` with
`body: string` (`src/types/edition.ts:27-28`), sourced from a single Sanity string field
(`src/sanity/lib/editions-mappers.ts:167`: `themeSection: { body: raw.themeSection?.body ?? '' }`),
and `ThemeArtists.tsx:28` renders it as one `<p>{themeSection.body}</p>`. There is structurally
never more than one `<p>` in `.body`, so `:last-child` can never diverge from the base `& p`
rule — it's a vestigial guard for a shape that doesn't exist here.
**recommendation:** Once merged with DS-1's `textStyle: 'prose'`, drop the nested `& p` /
`& p:last-child` selectors entirely and set `marginBottom: '0'` directly on the `body` slot (no
behavior change — it already only ever wraps one paragraph).
**effort:** trivial

### PROP-1 — `headerLabel` is dead flexibility

**file:line:** `src/components/ArtistsTable/ArtistsTable.tsx:17,24`
**axis:** prop-api
**risk tier:** no-op
**evidence:** `headerLabel?: string` defaults to `'Artists'`. Both call sites — `ThemeArtists.tsx:31-38`
and `artists/page.tsx:34-40` — omit it, so every current render uses the default; grepping
`<ArtistsTable` across `src/` turns up only these two sites, neither passing the prop.
**recommendation:** Drop the prop and hardcode `'Artists'` in the `colHeader` markup until a
second label is actually needed (YAGNI) — re-add it the day a caller needs it.
**effort:** trivial

### DS-2 — Hero's top offset re-derives `pageHero`'s formula instead of composing it

**file:line:** `src/app/(site)/editions/[year]/_components/Hero.recipe.ts:24,27`
**axis:** ds-hygiene
**risk tier:** no-op — extracting a shared value changes nothing visually
**evidence:** `Hero.recipe.ts` sets `paddingTop: '[calc(token(sizes.nav) + 80px)]'` (base) /
`'[calc(token(sizes.nav) + 120px)]'` (md) — byte-identical to the `pageHero` layerStyle's own
`paddingTop: { base: 'calc(token(sizes.nav) + 80px)', md: 'calc(token(sizes.nav) + 120px)' }`
(`src/design-system/tokens.ts:535-538`). The file's own docstring even names the coupling
("The top offset mirrors the shared `pageHero` layerStyle... so the frame starts at the same y
as the title on every other page") — it's a *known*, not accidental, duplication, but it's still
two independent places that must be edited in lockstep with nothing enforcing that.
**recommendation:** Extract the shared formula (e.g. a `layerStyles.heroTopOffset` fragment, or
a `sizes`/spacing token pair) that both `pageHero` and `hero` compose, so a future nav-height
change can't update one and silently miss the other.
**effort:** small

### ADV-1 — Tape's nav-tuck offset is an undocumented dependency on Navigation's logo geometry

**file:line:** `src/app/(site)/editions/[year]/_components/Hero.tsx:11-13`
**axis:** adversarial (hidden coupling)
**risk tier:** no-op (this is a documentation fix, not a value change)
**evidence:** `tapeTheme`'s `marginLeft` ramp (`{base:'[10px]', md:'[18px]', lg:'[-36px]',
xl:'[-40px]'}`) exists, per its own comment, to "tuck the theme tape under the nav" — i.e. it's
tuned to line up with `Navigation.recipe.ts`'s `logo` slot (`left: 'gutter'`, width stepping
`[40px]→touch(48px)→[56px]→[60px]` across the same breakpoints, `Navigation.recipe.ts:29-36`).
Nothing in either file cross-references the other, so the coupling is invisible: a future resize
of the nav logo would silently desync the tape with no type error, lint failure, or visual test
to catch it. (This *is* the floor rule's "alignment geometry that must stay in proportion"
exception, so the raw values are correctly kept — the gap is only that the "why these numbers"
is missing.)
**recommendation:** Add a one-line comment on `tapeTheme` naming `Navigation.recipe.ts`'s `logo`
slot as the value it's tuned against, so the next person resizing the nav knows to re-check here.
**effort:** trivial

### DS-3 — `tapes` bottom/maxWidth ramps are the one undocumented block in this file

**file:line:** `src/app/(site)/editions/[year]/_components/Hero.recipe.ts:80-89`
**axis:** ds-hygiene
**risk tier:** no-op (doc only — these are tuned against the photography, not safe to snap)
**evidence:** `tapes` steps `bottom: '[8%]' → '[10%]' → '[11%]'` and `maxWidth: '[94%]' → '[72%]'
→ '[62%]' → '[58%]'` across four breakpoints. Every other ramp/bracket in this file (the
`paddingTop` calc, the `grayscale(0.3)`, the `-1px` outline offset) has an inline rationale
comment; this one doesn't, breaking the file's own convention. `preset.ts:19-28`'s exception
list ("local scrims, the artists-table barcode, mask geometry, %-centering / hairline px") does
not explicitly name this pattern.
**recommendation:** These read as tuned art-direction against the hero photography (not
alignment-locked to another declared value, so token-snapping would just launder a magic number
into a magic name per the policy's own caveat) — keep them raw, but add the one-line "why" the
rest of the file already gives every other bracket.
**effort:** trivial

### DS-4 — Scattered undocumented raw px/% brackets

**file:line:** `ThemeArtists.recipe.ts:24` (`maxWidth: '[525px]'`), `artists/page.tsx:35`
(`maxWidth: '[820px]'`), `ArtistsTable.recipe.ts:76-77` (`minWidth: '[28px]'` / `'[32px]'`)
**axis:** ds-hygiene
**risk tier:** no-op (doc only)
**evidence:** None of these three recur elsewhere (grepped `525px`, `820px` — single hits each),
so the "second occurrence → token" trigger doesn't force a token yet, but `preset.ts:20-21`
still treats every bracket as "debt to justify or remove, not a blessed exception." `525px` /
`820px` are both narrow-column caps with no comment explaining the number (unlike the rest of
`ThemeArtists.recipe.ts`, which justifies everything else). `28px`/`32px` on `ArtistsTable`'s
`num` slot is plausibly load-bearing (fitting `padNum`'s 3-digit `tabular-nums` output without
wrap — a genuine "exact number matters" case) but that reasoning isn't written down either.
**recommendation:** One-line rationale comment on each; no value change. If a third narrow-column
width shows up later, that's the actual trigger to extract a shared `sizes` token.
**effort:** trivial
