## Detailed findings

### DEAD-1 — redundant `gridColumn` overrides duplicate the grid's own default
**file:line**: `ThemeArtists.recipe.ts:42` (`gridColumn: '1'`), `:55` (`gridColumn: '2'`)
**axis**: dead-code
**risk tier**: no-op
**smell**: Explicit column placement on the only two children of a two-column grid.

**evidence**: `inner` composes `editorialSplit()` (`src/design-system/recipes/editorial-split.ts:10-18`), which sets `display: 'grid'` with exactly two `gridTemplateColumns` tracks from `lg`. `ThemeArtists.tsx:26-38` renders exactly two children into `inner` — `body` then `<ArtistsTable>` — in that DOM order. CSS grid auto-placement puts item 1 in track 1 and item 2 in track 2 by default; `gridColumn: '1'`/`'2'` state exactly what would happen anyway.

**recommendation**: Delete both `gridColumn` declarations (keep `alignSelf: 'start'` on `body`). Removes 2 lines, zero visual change — the grid renders identically because the override equals the default.

**effort**: trivial

---

### DS-2 — `& p` block re-declares `textStyles.prose`
**file:line**: `ThemeArtists.recipe.ts:43-51`
**axis**: ds-hygiene
**risk tier**: small-visual
**smell**: A 6-property nested selector duplicating an existing textStyle almost field-for-field.

**evidence**: `textStyles.prose` (`src/design-system/tokens.ts:511-513`) = `{ fontFamily: 'body', fontSize: 'base', lineHeight: 'body', color: 'body' }`. The `& p` block declares the same four plus `fontWeight: 'regular'` (already the inherited default weight — no scale member overrides it upstream) and `textAlign: 'left'` (already the LTR default) — both no-ops restated as if load-bearing.

**recommendation**: Replace the block with `textStyle: 'prose'` plus the one value `prose` doesn't carry: `textWrap: '[pretty]'` (matches the bracketed convention already used for this exact keyword at `SectionHeading.recipe.ts:18`, `EditionCard.recipe.ts:85`, `Manifesto.recipe.ts:35` — not a new inconsistency). Collapses a 9-line block to 2 lines.

**effort**: trivial

---

### DS-4 — `outlineOffset: '[-1px]'` is unnamed and paired with a token it must track
**file:line**: `Hero.recipe.ts:55` (inside `frame._after`, alongside `outline: 'hairline'` at `:54`)
**axis**: ds-hygiene
**risk tier**: small-visual
**smell**: Bracketed literal whose only reason to be `-1px` is "inset flush with the 1px hairline it's drawn against" — a relationship expressed nowhere in code.

**evidence**: Every other `outlineOffset` in the codebase is the `xs` token (outset focus rings — `ExternalGallery.recipe.ts:60`, `Calendar.recipe.ts:118`, `Navigation.recipe.ts:85`, etc.) or hardcoded per-component focus values in `src/design-system/recipes/*`. None of those are a match for this inset case; `borders.hairline` (`tokens.ts:132-134`) is the only related token, and its `width` sub-field isn't independently addressable via `token()` for a `calc()` derivation.

**recommendation**: Add a new one-off token next to `borders.hairline` in `tokens.ts` (e.g. `sizes.hairlineInset: { value: '-1px' }`) and reference `outlineOffset: 'hairlineInset'`. Per the no-fourth-option rule, a single-use value still gets a name — a bracket-plus-comment isn't a legal substitute.

**effort**: trivial

---

### DS-5 — `grayscale(0.3)` is the third unnamed occurrence of the same value
**file:line**: `Hero.recipe.ts:46`
**axis**: ds-hygiene
**risk tier**: small-visual
**smell**: `grayscale(0.3)` ≡ `grayscale(30%)`, and `grayscale(30%)` is already bracketed independently at `Calendar.recipe.ts:167` and `EditionCard.recipe.ts:48`. Three files, one un-named value.

**evidence**: `grep -rn grayscale src/` turns up `filter: '[grayscale(30%) brightness(1)]'` (Calendar.recipe.ts:167) and `filter: '[grayscale(30%) brightness(1)]'` (EditionCard.recipe.ts:48) — the grayscale component of both is value-identical to Hero's. Well past the "second occurrence → token" bar in `preset.ts:26-27`.

**recommendation**: Add a token for the value (e.g. `tokens.assets.grayscaleSubtle: { value: 'grayscale(0.3)' }` alongside the existing `brushStrokeX`/`brushStrokeY` entries in `tokens.ts:163-166`) and reference it in Hero as `filter: 'token(assets.grayscaleSubtle)'` (the other two sites compose it with `brightness(1)` so aren't a 1:1 swap — out of scope here, but the token now exists for them).

**effort**: trivial

### DS-6 — `tapes.bottom` / `tapes.maxWidth` are two un-derived %-ramps
**file:line**: `Hero.recipe.ts:80` (`bottom: '[8%]'`), `:85` (`maxWidth: '[94%]'`), `:87-89` (md/lg/xl steps)
**axis**: ds-hygiene
**risk tier**: small-visual
**smell**: Two independent 3–4-step percentage ramps with no token, no comment on why these specific numbers, and no stated relationship to each other despite riding the same breakpoints in the same slot.

**evidence**: Existing precedent for exactly this shape already exists in the token file — `semanticTokens.sizes.nav` and `semanticTokens.spacing.gutter`/`gridGap` (`tokens.ts:276-279`, `311-320`) are `{ value: { base, md, lg, xl, … } }` stepped tokens for the same kind of art-directed geometry.

**recommendation**: Promote both ramps into named stepped tokens in `tokens.ts` (e.g. `spacing.heroTapeBottom`, `sizes.heroTapeMaxWidth`) using the identical shape as `sizes.nav`, then reference them in the recipe. Same numbers, now discoverable and greppable instead of two parallel brackets.

**effort**: small

### DS-7 — call-site `tapeTheme` marginLeft ramp is a magic 4-step number, comment names purpose not derivation
**file:line**: `Hero.tsx:11-13`
**axis**: ds-hygiene / adversarial
**risk tier**: small-visual
**smell**: `marginLeft: { base: '[10px]', md: '[18px]', lg: '[-36px]', xl: '[-40px]' }`, justified only by the comment "Container positioning only — tuck the theme tape under the nav." That names *what* the block does, not *why* these four numbers — exactly the axis-6 "comment where a refactor should be" smell.

**evidence**: The values aren't visibly derived from `semanticTokens.sizes.nav` (`60/72/84/100px`) despite the comment invoking the nav, and they don't match `spacing.gutter` (`16/40/80px`) either. `EditionTheme.recipe.ts:35` gives the tape itself `padding: '[0.45em 0.6em]'`, which scales with its `size: 'huge'` fontSize ladder (`Hero.tsx:42-43` passes `size="huge"`) at the same breakpoints — plausible source of the lg/xl negative pull-back, but not expressed in code.

**recommendation**: At minimum, tokenize the ramp (same `{base,md,lg,xl}` shape as `sizes.nav`) so the numbers are named in one place instead of a bracket ramp at the call site. If the negative lg/xl values are confirmed to be compensating for `EditionTheme`'s own `0.6em` padding, express the offset as a CSS var computed off that padding instead of an independent literal, so the two can't silently desync.

**effort**: small

### DS-8 — `inner.maxWidth: '[525px]'` has no token and no stated relationship
**file:line**: `ThemeArtists.recipe.ts:24`
**axis**: ds-hygiene
**risk tier**: small-visual
**smell**: Genuinely novel one-off value (no scale member close to 525px; not the `measure` prose-width token at `tokens.ts:158`, which is `60ch` and serves a different purpose), left as a bare bracket.

**evidence**: `grep -rn '525px' src/` returns only this one line — confirmed novel, not a missed reuse.

**recommendation**: Per rule 3 (a value that appears once still gets a name), add it to `tokens.sizes` in `tokens.ts` and reference the token. A bracket with no sibling to explain it is exactly the "load-bearing exception" this prompt's premise rules out.

**effort**: trivial

### DS-3 — Hero's `paddingTop` restates `layerStyles.pageHero` instead of composing it
**file:line**: `Hero.recipe.ts:24` (base), `:27` (md)
**axis**: ds-hygiene
**risk tier**: visual-risk
**smell**: The component's own docstring (`Hero.recipe.ts:10-11`) says the top offset "mirrors the shared `pageHero` layerStyle" — an admission that this is a duplicate, not a coincidence, followed by two bracketed `calc()` literals rather than a composition.

**evidence**: `layerStyles.pageHero.paddingTop` (`tokens.ts:535-538`) is `{ base: 'calc(token(sizes.nav) + 80px)', md: 'calc(token(sizes.nav) + 120px)' }` — byte-for-byte the same calc, same breakpoints, as Hero's `'[calc(token(sizes.nav) + 80px)]'` / `'[calc(token(sizes.nav) + 120px)]'`, just bracketed instead of composed.

**recommendation**: Compose `layerStyle: 'pageHero'` directly on the `hero` slot's base object, then keep only the property that genuinely differs — `paddingBottom` (Hero: `3xl`/`4xl`; pageHero: `2xl`/`3xl`) — as an explicit sibling declaration in the same object (Panda applies `layerStyle` first, so the explicit `paddingBottom` still wins). Deletes both bracketed calc lines; the "mirrors" claim becomes structurally enforced instead of asserted in a comment. Flagged `visual-risk` (not `no-op`) because composing `pageHero` also pulls in `background: 'black'` (≡ current `surface`, no change) and `color: 'white'` (Hero currently declares no `color` — likely a no-op since it has no direct text and children set their own colors, but worth an eyeball since it's an inherited-property change, not a literal-value swap).

**effort**: small
