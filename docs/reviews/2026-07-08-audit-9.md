Scope: `src/components/Carousel/`, `src/components/Lightbox/`, `src/components/Figure/`,
plus their shared DS machinery (`src/design-system/recipes/carousel.ts`,
`src/design-system/recipes/button.ts`, `src/design-system/tokens.ts`) and call

`src/design-system/preset.ts:15-27` confirms "no exceptions" and calls every
remaining `[bracketed]` literal migration backlog, not sanctioned art
direction — that premise still holds, no exception list has crept back in.

### STRUCT-1 — `HomepageCarousel.tsx:17,51-56`, `GalleryCarousel.tsx:25,70-75`, sibling `MediaKitStrip.tsx:16,48-56`

**axis:** structure

**smell:** All three current `Carousel` + `Lightbox` pairings repeat the exact
same four lines:

```tsx
const [lightboxIndex, setLightboxIndex] = useState<number | null>(null)
...
<Lightbox
  images={...}
  index={lightboxIndex}
  onClose={() => setLightboxIndex(null)}
  onIndexChange={setLightboxIndex}
/>
```

**evidence:** `HomepageCarousel.tsx:17` + `:51-56`, `GalleryCarousel.tsx:25` +
`:70-75`, and the out-of-scope sibling `src/app/(site)/press/_components/MediaKitStrip.tsx:16` + `:48-56` — identical state shape, identical `onClose`/`onIndexChange` closures, three times.

**recommendation:** extract a `useLightboxIndex()` hook (e.g. in
`src/components/Lightbox/`) returning `{ index, open, close, onIndexChange }`
(or `[index, setIndex]` plus the two derived closures). Each of the three call
sites collapses from ~6 lines of state + wiring to 2. Applies to
`HomepageCarousel.tsx`, `GalleryCarousel.tsx`, and also `MediaKitStrip.tsx` if
the follow-up extends beyond this review's scope. No rendered-output change.

### DS-1 — `design-system/recipes/carousel.ts:38-89`

**axis:** ds-hygiene

**smell:** `prevTrigger` and `nextTrigger` are byte-for-byte identical style
objects; `autoplayTrigger` differs only by omitting `_disabled`. All three are
also functionally the same shape as `button.ts`'s `icon` variant: `44px ×
44px`, transparent background, `color: 'heading'`, hover → `action`,
`_focusVisible` ring.

**evidence:** `carousel.ts:38-50` (`prevTrigger`) vs `:52-65` (`nextTrigger`)
vs `:77-89` (`autoplayTrigger`) — three separate slot declarations. Compare
`button.ts:60-70` (`icon` variant): same `44px` box, same `color: 'heading'`,
same `_hover: { color: 'action' }` shape (via `colorShift`), same
`_focusVisible` ring pattern used across both files.

**recommendation:** merge the three trigger slots into one `trigger` slot
class applied to all three controls (collapses 3 slot declarations → 1, ~35
lines → ~15), and have it compose `button({ variant: 'icon' })`'s token
values instead of re-declaring `44px`/color roles a second time in a
different file. If Ark's trigger components can't take Button's className
directly, at minimum share the `44px` box via one token
(`sizes.iconButton`) referenced from both `button.ts` and `carousel.ts`
instead of the value appearing four times across two files. No visual change
if the merge preserves current computed styles exactly.

### DS-2 — `Carousel.tsx:91-92`

**axis:** ds-hygiene

```tsx
padding={mode === 'rail' ? token('spacing.gutter') : '0px'}
```

### DS-3 — `HomepageCarousel.recipe.ts:18-23`

**axis:** ds-hygiene

```ts
vignette: {
  position: 'absolute',
  inset: '0',
  pointerEvents: 'none',
  background: '[radial-gradient(ellipse at center, transparent 55%, rgb(0 0 0 / 0.45) 100%)]',
```

recipe file (bracket replaced by a token reference of similar length).

### DS-4 — `GalleryCarousel.recipe.ts:3-9,13-27`

**axis:** ds-hygiene

**smell:** the file's own header comment says: "The bracketed clamp()/vh
pairs are the rail's per-breakpoint art direction" — exactly the kind of
"intentionally raw because X" comment the no-exceptions rule treats as a
claim to verify, not a settled fact. The `slide` slot ramps `width` and
`height` together across 7 breakpoints (base, a landscape-phone override, md,
lg, xl, 2xl, 4xl) as 14 raw bracket values.

**evidence:** the DS already has a template for exactly this shape —
`layerStyles.heroTapeOffset` (`tokens.ts:598-606`) bundles two properties
(`bottom` + `maxWidth`) that ramp together across breakpoints into one named
layerStyle, with a comment explaining why: "stepped in tandem across
breakpoints so the two ramps live in one named place instead of two parallel
brackets at the call site." Sibling raw ramp with the same shape (a rail
item's clamped width, stepped per breakpoint) also exists in
`src/app/(site)/press/_components/MediaKitStrip.recipe.ts:14,23-24` — a third
recurrence of "clamped rail-item width, ramped per breakpoint" (values
differ, but the shape is the same as `GalleryCarousel.recipe.ts`'s `slide`
width ramp).

**recommendation:** extract a `layerStyles.galleryRailFrame` (or similar)
bundling `width` + `height` across the same 7 breakpoints, the same way
`heroTapeOffset` bundles `bottom` + `maxWidth`. Collapses 14 inline bracket
values in `GalleryCarousel.recipe.ts` into one named, greppable definition
call-site: `slide: { layerStyle: 'galleryRailFrame' }`. The comment that
justified the brackets as "art direction" is then unnecessary — the layerStyle
name and its `tokens.ts` neighbor comment carry that context instead. Scoped
fix is `GalleryCarousel.recipe.ts` only; the `MediaKitStrip.recipe.ts:14,23-24`
ramp is a separate value family (different card, different numbers) and is
cited here only as a sibling shape, not folded into the same layerStyle.

### DS-5 — `GalleryCarousel.recipe.ts:40-46`

**axis:** ds-hygiene

```ts
'& img': {
  transition: '[transform {durations.entrance} {easings.expo}, filter {durations.reveal} {easings.quint}]',
  filter: '[brightness(0.9) contrast(1)]',
},
_hover: {
  '& img': { transform: 'scale(1.05)', filter: '[brightness(1) contrast(1.1)]' },
  ...
},
```

**smell:** a raw rest/hover filter pair — exactly the shape `tokens.assets`
already names for the "develop" treatment elsewhere (dim/desaturated at rest,
warms up on hover), but this pair reinvents it with different filter
functions instead of extending that family.

**evidence:** `tokens.ts:196-200` — `assets.developRest` / `assets.developHover`
is the existing named pair for "fully desaturated + dimmed at rest, warming
back to color + full brightness on hover", shared by `EditionCard` and
Calendar's run cards per its own comment. This gallery item filter
(`brightness(0.9) contrast(1)` → `brightness(1) contrast(1.1)`) is a *different*
visual (contrast, not grayscale) but structurally the same rest/hover asset
pair.

**recommendation:** add `assets.galleryDevelopRest` /
`assets.galleryDevelopHover` to `tokens.ts` next to the existing
`developRest`/`developHover`, and reference them in the recipe instead of two
inline brackets. The `transition` line's brackets are a different case —
they compose two *existing* tokens (`durations.entrance`/`easings.expo` and
`durations.reveal`/`easings.quint`) into one shorthand Panda's recipe API
can't express without string interpolation, so no further action there; that
bracket is derivation, not raw drift. No visual change from tokenizing the
filter values.

### DS-6 — `GalleryCarousel.recipe.ts:49-56`

**axis:** ds-hygiene

```ts
_before: {
  content: '""',
  layerStyle: 'gradientBorder',
  padding: '[2px]',
  transitionProperty: '[opacity]',
  transitionDuration: 'medium',
  transitionTimingFunction: 'quint',
},
```

**smell:** the `padding` (ring width) is a raw bracket, and the
`transitionProperty`/`transitionDuration`/`transitionTimingFunction` trio is
duplicated verbatim at the only other call site of this same layerStyle.

**evidence:** sibling at `src/app/(site)/editions/[year]/_components/Calendar/Calendar.recipe.ts:156-164`:
```ts
_before: {
  content: '""',
  layerStyle: 'gradientBorder',
  padding: '[0.5px]',
  transitionProperty: '[opacity]',
  transitionDuration: 'normal',
  transitionTimingFunction: 'quint',
},
```
Same shape, same two of three transition properties, different `padding`
(`2px` vs `0.5px`) and one different `transitionDuration` (`medium` vs
`normal`).

**recommendation:** fold `transitionProperty: 'opacity'` and
`transitionTimingFunction: 'quint'` — the two properties genuinely identical
across both call sites — into the `gradientBorder` layerStyle definition
itself (`tokens.ts:649-664`), leaving only `padding` (the ring width) and
`transitionDuration` as call-site deltas. That removes 2 duplicated lines
from each of the two call sites (net -4 lines) in exchange for +2 lines once
in `tokens.ts`. Separately, tokenize the two ring-width values instead of
leaving them bracketed — e.g. `borderWidths.gradientRing` (`2px`) and
`borderWidths.gradientRingHairline` (`0.5px`) — so a third call site finds a
named choice instead of picking a third arbitrary width. Scoped fix is
`GalleryCarousel.recipe.ts`; also-applies-to `Calendar.recipe.ts:156-164` if
the follow-up folds both call sites at once. No visual change.

### DS-7 — `Lightbox.tsx:163`

**axis:** ds-hygiene

```ts
transition: isDragging
  ? 'none'
  : 'transform var(--durations-normal) var(--easings-expo), opacity var(--durations-normal) var(--easings-quint)',
```

**smell:** the CSS custom-property names are hand-typed strings
(`var(--durations-normal)`, `var(--easings-expo)`) instead of going through
the `token()` helper, which is the sanctioned indirection for exactly this —
already imported and used for the same purpose one file over in
`Carousel.tsx:91` (`token('spacing.md')`).

**evidence:** `Carousel.tsx:8,91` shows the established pattern:
`import { token } from 'styled-system/tokens'` → `token('spacing.md')`. Panda
generates the `--durations-normal`/`--easings-expo`/`--easings-quint` var
names as an implementation detail of its token naming convention; hand-typing
them creates a silent breakage risk if that convention ever changes, whereas
`token()` calls are regenerated together with it.

**recommendation:** replace the two hand-written `var(--...)` pairs with
`token('durations.normal')`, `token('easings.expo')`, `token('easings.quint')`
composed into the same template string. No visual change — same resolved CSS
variable names today, just sourced through the indirection instead of
restated by hand.

### DS-8 — `Lightbox.tsx:217,228`

**axis:** ds-hygiene

```tsx
className={cx(s.nav, css({ left: '0', '--nav-nudge': '-2px' }))}
...
className={cx(s.nav, css({ right: '0', '--nav-nudge': '2px' }))}
```

**smell:** the hover-nudge distance is a raw, unnamed `2px`/`-2px` pair
declared independently at the two call sites, with no shared source — a
future change to one arrow's nudge distance can silently desync from the
other.

**evidence:** the DS already has the exact pattern for "one token, negate it
at the mirrored site" — `tokens.ts:122`:
```ts
hairlineOverlap: { value: 'calc({borderWidths.hairline} * -1)' },
```
used to negate `borderWidths.hairline` for the opposite direction instead of
hardcoding two independent literals.

**recommendation:** add a single token (e.g. `spacing.navNudge = '2px'`) and
derive the left arrow's value as `calc({spacing.navNudge} * -1)` — either as
a second token (`navNudgeNegative`) following the `hairlineOverlap` naming
convention, or inline via `calc(token(spacing.navNudge) * -1)` at the one
call site that needs the negative. Either way, one number is declared once;
the second is derived, not independently typed. No visual change.

### ADV-1 — `Lightbox.tsx:18`, `Lightbox.recipe.ts:33,36`

**axis:** adversarial (hidden coupling)

```ts
// Lightbox.tsx:18
const SIZES = '(min-width: 768px) calc(100vw - 160px), 90vw'
```
```ts
// Lightbox.recipe.ts:31-36
frame: {
  width: '[90vw]',
  height: '[85vh]',
  ...
  md: { width: '[calc(100vw - 160px)]', height: '[90vh]' },
},
```

**smell:** `next/image`'s `sizes` attribute and the frame's actual CSS width
independently restate the same two numbers (`90vw` at base, `calc(100vw -
160px)` at `md`/`768px`). Nothing derives one from the other — a change to
the frame's breakpoint width in the recipe silently stops matching the
`sizes` hint, which then causes the browser to fetch a wrongly-sized image
(not a visual bug, but a real regression class: over/under-fetching).

**recommendation:** define the two viewport-width expressions once — e.g. a
co-located constant `const FRAME_WIDTH = { base: '90vw', md: 'calc(100vw - 160px)' }`
consumed both by `SIZES` (built from it) and, if the recipe is later
refactored to accept the value some other way, by the recipe too. At minimum,
add a one-line assertion or shared constant so the two numbers have a single
declaration site instead of two hand-synced ones. This is a genuine
structural gap, not a Panda `strictTokens` violation (the `sizes` string
isn't a Panda-tracked value), which is why it's filed as adversarial rather
than ds-hygiene. No rendered-output change if the derived value matches
today's numbers exactly — flag for an eyeball only if the refactor changes
either number in the process.

### ADV-2 — `Carousel.tsx:32`, `Lightbox.tsx:39`

**axis:** adversarial (hidden coupling)

```ts
// Carousel.tsx:30-32
// Pointer travel (px) between pointerdown and click beyond which the click is
// treated as the tail of a mouse drag and suppressed.
const DRAG_CLICK_TOLERANCE = 8
```
```ts
// Lightbox.tsx:39
const AXIS_LOCK_THRESHOLD = 8
```

**smell:** both constants answer the same underlying question — "how many
pixels of pointer travel before this is a drag/swipe, not a static
tap/click?" — independently, in two different components, with two different
names and no shared source. They currently agree (`8`), but nothing enforces
that; a future tuning pass on one gesture has no reason to notice the other.

**recommendation:** extract a single named constant (e.g.
`POINTER_DRAG_TOLERANCE_PX = 8`) into a small shared gesture-utility module
(or re-export from one of the two components) and import it in both places.
Collapses two independent magic-number declarations into one. This is a JS
constant, not a Panda style value, so it sits outside the `strictTokens`
raw-value rule proper, but the underlying smell — two values that must stay
in sync by convention rather than by construction — is the same one that
rule exists to prevent.

### DEAD-1 — `Lightbox.tsx:157`

**axis:** dead-code (consistency)

```ts
const verticalProgress = Math.min(1, drag.y / 300)
```

**smell:** every other drag-gesture threshold in this file is a named

for the next reader tuning gesture feel.

### TYPES-1 — `Figure.tsx:31`, `FallbackImage.tsx:16`

**axis:** types

```ts
// Figure.tsx:28-31
type FigureProps = {
  image: ImageData | null | undefined
  sizes: string
} & Pick<ImageProps, 'className' | 'priority' | 'draggable' | 'style'>
```
```ts
// FallbackImage.tsx:16-21
type FallbackImageProps = Pick<ImageProps, 'className' | 'priority' | 'draggable' | 'style'> & {
  src: string
  alt: string
  sizes: string
  blurDataURL?: string | undefined
```

**smell:** the identical `Pick<ImageProps, 'className' | 'priority' |

future prop needs to join or leave that pass-through list, both files must be

in `Figure.tsx` (the parent of the pair), export it, and have
`FallbackImageProps` intersect it instead of re-picking. Collapses one
duplicated type expression into one declaration + one import. No behavior
change.

### PROP-1 — `Lightbox.tsx:13-16`, `HomepageCarousel.tsx:52`

**axis:** prop-api

```ts
// Lightbox.tsx:13-16
export interface LightboxImage {
  src: string
  caption: string
}
```
```tsx
// HomepageCarousel.tsx:51-52
<Lightbox
  images={images.map((image) => ({ src: image.src, caption: '' }))}
```

**smell:** `caption` is required, but the homepage hero has no caption data
(`HeroImage` — `types/edition.ts:11-15` — carries no caption field), so this
call site fabricates `''` purely to satisfy the type. The other two call
sites (`GalleryCarousel.tsx:23`, and the sibling `MediaKitStrip.tsx:49-52`)
both supply real caption text, so this isn't a case for deleting the prop —
just for making it reflect that captions are genuinely optional.

**evidence:** `Lightbox.tsx:211` already guards the render with
`{current.caption && <Eyebrow ...>}`, so an empty string already produces the
same (no-render) behavior an `undefined` would.

**recommendation:** change `caption: string` to `caption?: string | undefined`
(matching this codebase's `exactOptionalPropertyTypes` convention seen on
`CarouselProps.className`/`id`), and have `HomepageCarousel.tsx:52` map to
`{ src: image.src }` instead of fabricating `caption: ''`. Removes one
synthetic value at the one call site that doesn't have real data. No
rendered-output change (`''` and `undefined` both skip the caption render
today).
