# Component audit - `visit/`

**Scope:** `src/app/(site)/visit/page.tsx`,
`src/app/(site)/visit/_components/{VisitSection,VenuesView,VisitFaq}.{tsx,recipe.ts}`.

**Raw-value policy used:** `panda.config.ts:12-13` has `strictTokens: true`.
`src/design-system/preset.ts:15-28` says every style value is a token or a
deliberate bracketed exception; recurring values become tokens or composition;
most bracketed literals are migration backlog; snap to the nearest token or
refactor/recompose unless the value is load-bearing geometry, local scrim/mask
art direction, percent-centering, or hairline pixels.

## Detailed Findings

### DS-1 - FAQ answer retypes `prose`

- **file:line:** `src/app/(site)/visit/_components/VisitFaq.recipe.ts:15`
- **axis:** ds-hygiene
- **risk tier:** no-op
- **smell:** `answer` repeats the `prose` textStyle property-for-property.
- **evidence:** `VisitFaq.recipe.ts:16-19` sets `fontFamily: 'body'`,
  `fontSize: 'base'`, `lineHeight: 'body'`, `color: 'body'`. That is exactly
  `textStyles.prose` in `src/design-system/tokens.ts:499-500`.
- **recommendation:** use `textStyle: 'prose'` on `answer` and keep only
  `whiteSpace: 'pre-line'` locally.
- **effort:** trivial

### DS-2 - event/child meta labels drift from `metaLabel`

- **file:line:** `src/app/(site)/visit/_components/VenuesView.recipe.ts:92`,
  `src/app/(site)/visit/_components/VenuesView.recipe.ts:118`
- **axis:** ds-hygiene
- **risk tier:** small-visual
- **smell:** `eventWhen` and `childType` locally retype the same small uppercase
  muted label role.
- **evidence:** `eventWhen` and `childType` set body font, `2xs`, uppercase,
  `letterSpacing: 'label'`, and `color: 'muted'` (`VenuesView.recipe.ts:93-98`,
  `:119-123`). `textStyles.metaLabel` already owns that role plus
  `fontWeight: 'semibold'` (`src/design-system/tokens.ts:470-478`), and the
  home page uses it for edition metadata (`src/app/(site)/page.recipe.ts:195`).
- **recommendation:** set `textStyle: 'metaLabel'`; keep
  `fontVariantNumeric: 'tabular-nums'` on `eventWhen`. The only visual delta is
  the semibold weight, which fits the token-snapping rule.
- **effort:** trivial

### DS-3 - FAQ width bypasses the `measure` token

- **file:line:** `src/app/(site)/visit/_components/VisitFaq.recipe.ts:14`
- **axis:** ds-hygiene
- **risk tier:** small-visual
- **smell:** the FAQ list uses a bracketed pixel max-width with no local
  justification.
- **evidence:** `list: { maxWidth: '[760px]' }`; the design system has
  `sizes.measure: '60ch'` for "comfortable prose measure" (`tokens.ts:157-158`),
  and existing long-form/prose components use `maxWidth: 'measure'`
  (`PageHero.recipe.ts:24`, `PillarGrid.recipe.ts:38`, `privacy/page.recipe.ts:15`).
- **recommendation:** snap the accordion list to `maxWidth: 'measure'`, or add a
  local comment explaining why accordion disclosure width must stay exactly
  760px. Silent bracket is debt under the repo policy.
- **effort:** trivial

### DS-4 - decorative pixels bypass strict tokens

- **file:line:** `src/app/(site)/visit/_components/VisitSection.tsx:31`
- **axis:** ds-hygiene
- **risk tier:** visual-risk
- **smell:** the pixel decoration is a mini design system in a JS array, with
  raw px offsets/sizes applied through React `style`.
- **evidence:** `PIXELS` contains `'-12px'`, `'60px'`, `24`, `'-16px'`, etc.
  (`VisitSection.tsx:31-39`) and applies them as inline `top`/`left`/`width`/
  `height`/`background` (`VisitSection.tsx:66-74`). Because this is not Panda
  style authoring, `strictTokens` cannot police it. Some of these values may be
  deliberate art-direction geometry, but the current code does not say which
  relationships are load-bearing.
- **recommendation:** move the decoration into recipe-owned variants or CSS
  custom properties whose values are bracketed and justified in the recipe
  comment. If the exact offsets are not load-bearing, snap sizes/offsets to the
  spacing scale and keep only the truly proportional overlaps raw.
- **effort:** small

### STRUCT-1 - VisitSection is several components under one slot recipe

- **file:line:** `src/app/(site)/visit/_components/VisitSection.recipe.ts:17`
- **axis:** structure
- **risk tier:** visual-risk
- **smell:** one slot recipe owns the page shell, image art, facts, amenities,
  transport rows, and CTA.
- **evidence:** the recipe declares 23 slots (`VisitSection.recipe.ts:17-40`).
  The JSX then renders distinct blocks: image frame plus pixels
  (`VisitSection.tsx:58-77`), info blocks (`:84-99`), amenity strip (`:101-111`),
  transport list (`:113-124`), and CTA (`:126-133`). These are independent
  layout objects, not just anatomy slots of one component.
- **recommendation:** split the boundary at the visible blocks:
  `VisitImageFrame` owns frame/pixel art, `VisitInfoSummary` owns address/hours,
  `AmenityStrip` owns amenity icon chips, and `TransportList` owns route rows.
  Keep `VisitSection` as the composition shell with `sectionInner` and split
  layout.
- **effort:** involved

### API-1 - Accordion trigger accepts headings that become invalid button content

- **file:line:** `src/app/(site)/visit/_components/VenuesView.tsx:42`,
  `src/app/(site)/visit/_components/VisitFaq.tsx:30`
- **axis:** prop-api
- **risk tier:** behavioral
- **smell:** local call sites pass real heading elements into a trigger slot
  that the shared Accordion renders inside an Ark button trigger.
- **evidence:** `Accordion` renders `{item.trigger}` inside
  `<ArkAccordion.ItemTrigger>` (`src/components/ui/Accordion/Accordion.tsx:49-50`).
  `VenuesView` passes `<h4>{venue.name}</h4>` and `VisitFaq` passes
  `<h3>{entry.question}</h3>`. A button cannot contain heading flow content, so
  the DOM model is invalid even if it looks fine.
- **recommendation:** change the Accordion API so heading semantics are owned
  intentionally, for example `triggerLabel` plus `triggerHeadingLevel`, or make
  the trigger text a span and place any section heading outside the button. Do
  not pass heading nodes through a generic `trigger: ReactNode` slot.
- **effort:** small

### BEHAV-1 - directions CTA renders with an empty `href`

- **file:line:** `src/app/(site)/visit/_components/VisitSection.tsx:126`
- **axis:** adversarial
- **risk tier:** behavioral
- **smell:** the component always renders "Get Directions" even when there is no
  map URL.
- **evidence:** `mapsUrl` falls back to `''` (`VisitSection.tsx:46`) and the CTA
  anchor always renders with `href={mapsUrl}` (`VisitSection.tsx:126-128`).
  The route fallback passes `<VisitSection />` with no data
  (`src/app/(site)/visit/page.tsx:22`), so the empty link is reachable in the
  component's own documented usage.
- **recommendation:** render the CTA only when `mapsUrl` is truthy. If fallback
  UI should expose a placeholder, make it non-interactive.
- **effort:** trivial
