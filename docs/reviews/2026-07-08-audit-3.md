## Detailed findings (most-actionable first)

### DS-1 — `venueName` / `venueParent` duplicated verbatim across recipes
- **file:line**: `Calendar.recipe.ts:519-531`, `EventModal.recipe.ts:86-98`
- **axis**: ds-hygiene · **risk**: no-op · **effort**: small
- **smell**: The two inner venue slots are byte-for-byte identical in both recipes.
- **evidence**: Both declare
  `venueName: { color: 'gray.300', textTransform: 'uppercase', letterSpacing: 'subtle', fontWeight: 'medium' }`
  and
  `venueParent: { color: 'muted', textTransform: 'uppercase', letterSpacing: 'label', fontSize: '2xs', _before: { content: '"↳ "' } }`.
  Only the wrapping `venue` slot differs (Calendar `fontSize: 'xs'`, EventModal `fontSize: 'sm'` +
  `marginTop: 'sm'`). The JSX is duplicated too (`Calendar.tsx:294` `VenueLine` vs
  `EventModal.tsx:79-82`).
- **recommendation**: Extract a shared `VenueLine` component with its own co-located recipe;
  pass the size delta (`xs` vs `sm`) as a prop or `className` override. Removes 2 identical slot
  pairs and one inlined markup copy. See STRUCT-1 — same extraction.

### DS-2 — `recapFollowLabel` / `followLabel` re-declare the `metaLabel` textStyle
- **file:line**: `Calendar.recipe.ts:545-552`, `ComingSoon.recipe.ts:68-75`
- **axis**: ds-hygiene · **risk**: no-op · **effort**: trivial
- **smell**: Both slots hand-write the six properties that `textStyle: 'metaLabel'` already is.
- **evidence**: `metaLabel` (`tokens.ts:470`) = `{ fontFamily: 'body', fontSize: '2xs',
  textTransform: 'uppercase', letterSpacing: 'label', fontWeight: 'semibold', color: 'muted' }`.
  `recapFollowLabel` and `followLabel` declare exactly that set, nothing more. `metaLabel` is an
  established target (already used at `Calendar.recipe.ts:496`, `page.recipe.ts:195`,
  `VenuesView.recipe.ts:93,115`).
- **recommendation**: Replace both slot bodies with `{ textStyle: 'metaLabel' }`. Zero visual change.

### DS-3 — identical chip-list block declared three times
- **file:line**: `Calendar.recipe.ts:532` (`chips`), `EventModal.recipe.ts:75` (`types`),
  `CalendarFilters.recipe.ts:68` (`chips`)
- **axis**: ds-hygiene · **risk**: no-op · **effort**: trivial
- **smell**: The same horizontal-wrap list surface is retyped in three recipes.
- **evidence**: All three are `{ listStyle: 'none', display: 'flex', flexWrap: 'wrap', gap: 'sm' }`
  (EventModal's adds only `marginTop: 'sm'`). This is the exact pattern a `layerStyle` is for.
- **recommendation**: Add a small `chipRow` (or `tagList`) `layerStyle` and compose it; keep the
  EventModal `marginTop` at the slot. Low value on its own but folds cleanly into STRUCT-1 (the
  shared `TypeChips` would own this once).

### STRUCT-1 — EventModal re-inlines `VenueLine` + `TypeChips` instead of reusing them
- **file:line**: `EventModal.tsx:69-82` vs `Calendar.tsx:282-301`
- **axis**: structure · **risk**: no-op · **effort**: small
- **smell**: `Calendar.tsx` already defines `TypeChips` and `VenueLine`; the modal reimplements
  both by hand (map `event.types` → `Badge tone="outline"`, and the `venueName`/`venueParent`
  span pair).
- **evidence**: `EventModal.tsx:70-77` reproduces `TypeChips`' body; `EventModal.tsx:79-82`
  reproduces `VenueLine`'s body. Both currently read Calendar's shared recipe instance `s`, so
  they can't simply be imported without pulling the whole `calendar()` recipe.
- **recommendation**: Promote `TypeChips` and `VenueLine` out of `Calendar.tsx` into their own
  co-located components (with the shared recipe from DS-1/DS-3). Row, run card, and modal all
  consume them. This is the extraction boundary that resolves DS-1 and DS-3 at once: `VenueLine`
  owns venue name/parent, `TypeChips` owns the badge list.

### STRUCT-2 — "Follow …" block duplicated as JSX and recipe across two files
- **file:line**: `Calendar.tsx:111-126` + `Calendar.recipe.ts:544-553` vs
  `ComingSoon.tsx:40-55` + `ComingSoon.recipe.ts:67-76`
- **axis**: structure · **risk**: visual-risk (shared markup change) · **effort**: small
- **smell**: A label + `<ul>` of `<Button asChild variant="link">` social links exists twice,
  down to the recipe slots (`recapFollow`/`recapFollowLabel`/`recapLinks` ≈
  `follow`/`followLabel`/`links`).
- **evidence**: The `socials.map(...)` bodies are identical; `recapLinks` and `links` are both
  `{ listStyle: 'none', display: 'flex', flexWrap: 'wrap', gap: 'md' }`; the label slots collapse
  under DS-2. The coupling is already implicit — `Calendar.tsx:19` imports `SocialLink` **from
  ComingSoon**.
- **recommendation**: Extract a `FollowLinks({ label, socials })` component and move the
  `SocialLink` type to a neutral module (or `calendar-filters.ts`/a local `types.ts`) so neither
  component owns the other's type. Both the recap and the coming-soon notice render it.

### DS-4 — `runName` and `eventName` are near-duplicate slots
- **file:line**: `Calendar.recipe.ts:232-239` vs `417-424`
- **axis**: ds-hygiene · **risk**: small-visual · **effort**: trivial
- **smell**: Two title slots share five properties and differ only in `color`.
- **evidence**: Both are `{ fontFamily: 'body', fontWeight: 'bold', fontSize: 'base',
  lineHeight: 'tight', letterSpacing: 'tight' }`; `runName` `color: 'gray.200'`, `eventName`
  `color: 'white'`. (Whether the color difference is intentional is worth a glance — the run card
  title reading dimmer than the agenda title looks incidental.)
- **recommendation**: Either unify on `white` and use one slot, or extract a `boardTitle`
  textStyle/layerStyle for the shared five and keep the `color` at each slot. Confirm the
  gray.200-vs-white distinction is deliberate before collapsing.

### STRUCT-3 — board header `meta` block duplicated across Calendar and ComingSoon
- **file:line**: `Calendar.recipe.ts:92-107` + `Calendar.tsx:93-101` vs
  `ComingSoon.recipe.ts:30-45` + `ComingSoon.tsx:26-30`
- **axis**: structure · **risk**: visual-risk · **effort**: small
- **smell**: `meta` (year • dot • label) and its `metaYear`/`metaDot` slots are duplicated;
  `ComingSoon.recipe.ts:4` even documents that it "mirrors the live calendar's header."
- **evidence**: `metaYear` (`color: 'highlight'`) and `metaDot`
  (`{ width: '[4px]', height: '[4px]', background: 'gray.700', borderRadius: 'circle' }`) are
  identical in both recipes; `meta` differs only by Calendar's `color: 'body'`. The `<SectionHeading
  flush>Calendar</SectionHeading>` + meta paragraph markup is the same shape in both components.
- **recommendation**: Extract a `CalendarBoardHeader` (title + meta row, with the trailing chunk
  as `children`/props) shared by `Calendar` and `ComingSoon`. Note this is a real structural change
  to two components — sequence it after the no-op wins.

### DS-5 — `--marker-col` ramp is non-monotonic and unexplained
- **file:line**: `Calendar.recipe.ts:76-79`
- **axis**: ds-hygiene · **risk**: small-visual · **effort**: small
- **smell**: The custom-prop ramp climbs `84 → 132 → 168` then **drops to 144** at `xl` and back
  to `160` at `2xl`, with only a one-line "scales up with the type" comment that the dip
  contradicts.
- **evidence**: `'--marker-col': '84px'`, `md 132px`, `lg 168px`, `xl 144px`, `2xl 160px`. A
  monotonic "scales with type" ramp would not dip at `xl`. Either it's compensating for the marker
  font-size step (`markerDay` is `3xl` base, `2xl` at `md`) and is deliberate, or it's a stale
  value. These are proportion/layout widths (marker column vs. the day glyph), so they may stay
  raw — but the **rationale must be written down** per the magic-ramp rule.
- **recommendation**: Verify the `xl` dip is intentional (screenshot at 1280px); then either fix
  it or add a comment explaining the marker-glyph coupling. Do not tokenize blindly.

### ADV-1 — modal body padding hardcodes the Back button's height
- **file:line**: `EventModal.recipe.ts:59-60`
- **axis**: adversarial · **risk**: visual-risk · **effort**: involved
- **smell**: `paddingTop: '[calc(36px + token(spacing.sm) + token(spacing.lg))]'` — the `36px` is
  the floating control bar's height (`controls` at `top: 'sm'`), i.e. the rendered height of the
  `<Button size="sm">`. Nothing enforces the two stay equal.
- **evidence**: The comment says "clears the floating control bar (36px + offset)". If the `sm`
  Button height changes in the Button recipe, the body content silently slides under or leaves a
  gap. `36px` is not a token and appears only here.
- **recommendation**: Prefer a layout that doesn't measure the button by hand — e.g. make the
  control bar participate in flow (a non-absolute header row) so the body's `paddingTop` follows
  from `gap`, or expose the control height as a shared token both the button and this calc read.
  At minimum, tie `36px` to whatever the Button `sm` min-height token is. Behavioral-adjacent;
  eyeball after.

### DS-6 — `metaDot` `[4px]` literal, duplicated
- **file:line**: `Calendar.recipe.ts:107`, `ComingSoon.recipe.ts:45`
- **axis**: ds-hygiene · **risk**: small-visual · **effort**: trivial
- **smell**: A 4px decorative dot with a raw literal, written twice.
- **evidence**: `{ width: '[4px]', height: '[4px]', ... }` in both. There is no 4px `sizes` token
  (only `spacing.xs: 4px`), so a clean snap isn't available for a `width`/`height`. Its second
  occurrence is the real trigger.
- **recommendation**: Fold it into the shared header (STRUCT-3) so the dot is declared once; that
  removes the duplication without needing a new size token. If STRUCT-3 is deferred, leave the
  bracket (it's a hairline-scale decorative dot) but note it's on the backlog.

## Clean axes

- **Types**: clean. `calendar-filters.ts` uses precise unions (`FilterSelection = string[] | null`,
  tri-state `showPast: boolean | null`); no `any`/unsafe `as`/`!`; the `CalendarFilters as Filters`
  alias in `CalendarFilters.tsx:11` is a legit name-clash rename. `exactOptionalPropertyTypes` /
  `noUncheckedIndexedAccess` respected (e.g. `byDay.get(iso) ?? []` at
  `calendar-filters.ts:282`).
- **Prop / API**: clean. `useShareLink(resolveUrl)` reads the URL at click time (documented);
  `RoutedEventModal` cleanly separates route/close concerns from the presentational `EventModal`;
  `HashScroller` is a focused effect with a correct dep + a docstring justifying its mount point.
- **Dead code**: clean. Every declared slot in all four recipes is applied in its component; no
  unreferenced exports/handlers; `shareIcon`/`shareCopied` are the shared dress for both share
  buttons and both are used.
