## Detailed findings

### ADV-1 — `EditionCard` re-derives `venue` by splitting a string whose structured source is already fetched
**file:line**: `EditionCard.tsx:12-15` (the `EditionCardData` doc comment), `:52` (`const [date, venue] = edition.dateTape.split(' · ')`)
**axis**: adversarial
**risk**: no-op (the split output and `venueLine` are byte-identical today — see evidence)
**smell**: The type comment claims `venueLine` isn't needed because `dateTape` already composes it — that claim is the justification for a fragile string-split that shouldn't need to exist.

**evidence**: `composeDateTape` (`src/lib/edition-dates.ts:44-53`) builds `dateTape` as `` `${range} · ${raw.venueLine}` `` — i.e. the venue half of the split is always exactly `raw.venueLine`, verbatim. `EDITION_CARDS_QUERY` (`src/sanity/lib/queries.ts:258-269`) already fetches `venueLine` (`:265`). But `mapEditionCard` (`src/sanity/lib/editions-mappers.ts:132-141`) never puts it on the returned object — only `dateTape: composeDateTape(raw)` survives. `EditionCard.tsx`'s `EditionCardData` (a `Pick<Edition, ...>`) follows suit and omits `'venueLine'`, even though `Edition` already carries it (`src/types/edition.ts:195`) for exactly this purpose ("kept alongside the composed `dateTape`" — `:191`). The component is then forced to reverse-engineer the venue substring via `.split(' · ')` — a private formatting contract owned by `composeDateTape`, invisible to the type system. The same fragile split recurs independently at `src/app/(site)/editions/[year]/opengraph-image.tsx:42` (`edition?.dateTape.split(' · ')[0]`), confirming this is a pattern, not a one-off.

**recommendation**: Add `venueLine: raw.venueLine ?? ''` to `mapEditionCard`'s return (mirroring `mapEdition`'s existing handling at `editions-mappers.ts:157`), add `'venueLine'` to `EditionCard.tsx`'s `EditionCardData` Pick, and replace `edition.dateTape.split(' · ')` with `edition.venueLine` for the venue span (keep splitting `dateTape` only for the date portion, matching the pattern `opengraph-image.tsx` already uses safely for date-only extraction). Deletes the dependency on a private join convention for the one piece of data that actually needs structure; net line count is flat (a field addition, a split simplification) but removes a hidden coupling entirely. No visual change — the rendered string is identical.

### DS-1 — media scrim gradient has no token, though the DS already has a home for named gradients
**file:line**: `EditionCard.recipe.ts:29-37` (`_after.background`, line 34 for the literal)
**axis**: ds-hygiene
**risk**: no-op
**smell**: `'[linear-gradient(180deg, rgb(0 0 0 / 0.5), transparent 30%, rgb(0 0 0 / 0.55))]'` is a raw bracketed literal with no sibling anywhere in `src/` (checked `rgb(0 0 0 / 0.5)`, `/0.55`, `/0.45`, `/0.6` — the only other 180deg gradient, `Manifesto.recipe.ts:49`, is the unrelated brush-stroke motif; `HomepageCarousel.recipe.ts:22`'s radial scrim uses different stops/opacity).

**evidence**: `tokens.gradients` (`src/design-system/tokens.ts:232-241`) exists as a category with exactly one member today, `heroVignette` — a named "photographic vignette for legibility" gradient. This scrim serves the identical purpose (legibility of the year badge over card imagery) but was never given the same treatment.

**recommendation**: Add a second member to `tokens.gradients` (e.g. `cardScrim`) with this exact value, and reference `background: 'token(gradients.cardScrim)'` at the call site. Per the no-fourth-option rule this applies even though the value is currently single-use — a named token beats an inline bracket. No visual change.

### DS-2 — develop-filter rest/hover pair is byte-identical to `Calendar.recipe.ts`, and the hover half re-types an existing token
**file:line**: `EditionCard.recipe.ts:39-51` (rest at `:42`, hover at `:48`)
**axis**: ds-hygiene
**risk**: no-op
**smell**: Two files declare the exact same "develop" filter treatment as independent unnamed literals; one of them (`grayscale(30%)`) is a percentage re-typing of a value the DS already named.

**evidence**: `filter: '[grayscale(100%) brightness(0.7)]'` (`:42`) is byte-identical to `Calendar.recipe.ts:190`. `filter: '[grayscale(30%) brightness(1)]'` (`:48`) is byte-identical to `Calendar.recipe.ts:167`. `grayscale(30%)` ≡ `grayscale(0.3)`, which is already `tokens.assets.grayscaleSubtle` (`tokens.ts:172`) — a prior audit (`2026-07-08-audit-5.md`, DS-5) added that token specifically because this value recurred, and explicitly deferred fixing these two `brightness`-paired sites as "out of scope." This is that deferred follow-up.

**recommendation**: Add two more `assets` members alongside `grayscaleSubtle` — e.g. `assets.developRest: 'grayscale(1) brightness(0.7)'` and `assets.developHover: 'grayscale(0.3) brightness(1)'` (the second literally `token(assets.grayscaleSubtle)`'s value plus `brightness(1)`) — and reference them via `token()` at both `EditionCard.recipe.ts:42,48` and `Calendar.recipe.ts:190,167`. Collapses 4 independently-typed literals (2 files × rest/hover) into 2 named tokens referenced 4 times. No visual change.

### DS-3 — `marginTop: '[-3rem]'` has no scale member, and the nearest numeric match is a different semantic
**file:line**: `EditionCard.recipe.ts:58-66` (`content` slot, literal at `:64`)
**axis**: ds-hygiene
**risk**: no-op
**smell**: The card's image-bleeds-under-content overlap is a raw, unnamed, unique value.

**evidence**: `-3rem` = `-48px`. The only existing token at that magnitude is `sizes.touch` (`tokens.ts:156`, `48px`) — named and documented as the WCAG touch-target minimum. Reusing it here would be exactly the "semantic-token misuse" this axis warns against (a token borrowed only because the number coincides, not because the concept matches). `spacing` has no `48px`-ish member either (`xs`/`sm`/`lg`/`xl` — nearest is `lg`'s clamp ceiling, not a flat value). No other file in `src/` uses a matching negative-margin overlap (checked `marginTop: '[-` project-wide — this is the only hit).

**recommendation**: Add a new `spacing` (or `sizes`) token for this value (e.g. `spacing.cardOverlap: { value: '3rem' }`) and reference it as `marginTop: '-cardOverlap'` (Panda's negative-token shorthand), dropping the bracket. Single-use today, but per the no-fourth-option rule that's not a reason to leave it raw. No visual change.

### DS-4 — no `opacity` token category exists anywhere in the DS; this is one of ~13 unnamed literals for the same intent
**file:line**: `EditionRailCard.recipe.ts:46-58` (`upcoming` variant; `opacity` at `:49`, `filter` at `:50`)
**axis**: ds-hygiene
**risk**: small-visual (if snapped to the nearest existing convention rather than minted as its own value)
**smell**: `tokens.ts` has no `opacity` scale at all, so every "muted/disabled" opacity in the codebase is independently invented.

**evidence**: `grep -rn "opacity: '0\." src/` turns up raw literals across at least 8 files: `0.42` ×2 (`Calendar.recipe.ts:173,280`, past-event de-emphasis), `0.45` (`LinkList.recipe.ts:22`, `aria-disabled`), `0.5` ×4 (`button.ts:27`, `carousel.ts:49,63`, plus checkbox's `0.5`/`0.6`/`0.8` hover ladder), `0.58` (this file), `0.85` ×2 (`EditionTheme.recipe.ts:55`, `Manifesto.recipe.ts:53` — a different, decorative intent, see DS-5). `0.5` is the closest thing to an existing convention for "disabled" (4 independent sites). `filter: '[grayscale(1)]'` (`:50`) is likewise unnamed; the DS already has `assets.grayscaleSubtle` (`grayscale(0.3)`) as a named partial-desaturation tier but no named full-desaturation counterpart.

**recommendation**: Add an `opacity` token category to `tokens.ts` (e.g. `disabled: { value: '0.5' }`, matching the 4-site majority convention) and either snap this call site to it (accept the small delta from 0.58→0.5, per the explicit allowance in the no-exceptions rule) or add a second member (e.g. `muted: '0.58'`) if the "still visible but non-interactive" tier is meant to read differently from the button/carousel "fully disabled" tier — that's a design call, not a mechanical one, but leaving it as a bare untokenized literal isn't an option either way. Add `assets.grayscaleFull: { value: 'grayscale(1)' }` alongside `grayscaleSubtle` and reference it here. If the opacity is snapped to `0.5`, this is small-visual; the grayscale token swap is no-op (exact value match).

### DS-5 — brush-stroke rule duplicates `Manifesto.recipe.ts` near-verbatim, differing only by axis
**file:line**: `EditionTheme.recipe.ts:45-56` (`root._before`)
**axis**: ds-hygiene
**risk**: no-op
**smell**: The same "brush-stroke rule fading from solid to transparent along one edge" motif is declared twice, independently, as a horizontal and a vertical variant of one idea.

**evidence**: `EditionTheme.recipe.ts:45-56`:
```
_before: {
  content: '""', position: 'absolute', top: '0', left: '0', right: '0',
  height: 'brushStroke',
  background: '[linear-gradient(90deg, token(colors.brushStroke) 0%, token(colors.brushStroke) 72%, transparent 100%)]',
  clipPath: 'token(assets.brushStrokeX)',
  opacity: '0.85',
}
```
vs. `Manifesto.recipe.ts:41-51`:
```
_before: {
  content: '""', position: 'absolute', left: '0', top: '0', bottom: '0',
  width: 'brushStroke',
  background: '[linear-gradient(180deg, token(colors.brushStroke) 0%, token(colors.brushStroke) 72%, transparent 100%)]',
  clipPath: 'token(assets.brushStrokeY)',
  opacity: '0.85',
}
```
Same `content`/`position` boilerplate, same gradient stop percentages, same `opacity: '0.85'` literal, same `token(colors.brushStroke)` pair — the only real differences are which two of {top,left,right,bottom} are pinned, `height` vs `width`, `90deg` vs `180deg`, and `brushStrokeX` vs `brushStrokeY`.

**recommendation**: Extract one shared `layerStyle` (e.g. `brushStrokeRule`) carrying the common `content`/`position`/gradient-stops/`opacity`, parameterized by axis (either two thin layer styles, `brushStrokeRuleX`/`brushStrokeRuleY`, each composing the shared base plus their 3 axis-specific properties, or one layerStyle plus a small per-call-site override for the 3 differing properties). Collapses ~18 duplicated lines across 2 files to one shared declaration + 2 short overrides. No visual change.

### DS-6 — `size` variants restate identical `aspectRatio` values twice, differing only at one breakpoint
**file:line**: `EditionCard.recipe.ts:116-123`
**axis**: ds-hygiene
**risk**: no-op
**smell**: `lg` and `md` size variants declare the same `base`/`md` aspect ratios (`16 / 9`, `21 / 9`) independently; only the `lg` breakpoint value differs (`21 / 9` vs `16 / 10`).

**evidence**:
```
lg: { media: { aspectRatio: { base: '16 / 9', md: '21 / 9', lg: '21 / 9' } } },
md: { media: { aspectRatio: { base: '16 / 9', md: '21 / 9', lg: '16 / 10' } } },
```
Two of the three breakpoint values are duplicated verbatim across both branches.

**recommendation**: Hoist the shared `{ base: '16 / 9', md: '21 / 9' }` into the slot's `base.media.aspectRatio`, and reduce each `size` variant to only the differing key: `lg: { media: { aspectRatio: { lg: '21 / 9' } } }`, `md: { media: { aspectRatio: { lg: '16 / 10' } } }`. Panda merges variant styles onto base per breakpoint key, so the computed result is unchanged — removes 2 duplicated literals. No visual change.

### PROP-1 — `&[class]` specificity hack overrides `EditionTheme`'s own properties instead of the rail getting a variant
**file:line**: `EditionRailCard.recipe.ts:20-37`
**axis**: prop-api
**risk**: no-op (recommendation preserves the exact same computed styles, only relocates ownership)
**smell**: An ad-hoc `className`-reaching override (`&[class]`) that should be a recipe variant on the component being overridden — the axis-4 checklist's canonical example, verbatim.

**evidence**: The comment at `:26-28` justifies the hack explicitly: *"`&[class]` (not `!`) bumps specificity so these reliably beat `EditionTheme`'s own classes for the same properties, instead of the outcome depending on generated CSS order."* That's the comment conceding the real problem (cascade-order fragility) and then working around it with a specificity trick rather than fixing the ownership. `EditionTheme.recipe.ts` already has exactly the extension point for this — its `size` variant (`huge`/`large`/`normal`, `:102-116`) is the named fontSize ladder; the rail needs a 4th rung ("two steps up from `normal`," per the rail's own comment at `:30-31`) plus its own padding treatment (`paddingTop: '[0.8em]'` at `:25`, `paddingLeft: '0'` at `:36`) — none of which `EditionTheme` currently exposes as a variant.

**recommendation**: Add a 4th `size` step to `EditionTheme.recipe.ts` (e.g. `rail`) carrying this exact fontSize ladder, and fold the rail-specific padding treatment into that same variant (or a small paired `dense`/compact-padding variant) so `EditionRailCard.tsx` passes `size="rail"` instead of the recipe reaching in via `&[class]`. Deletes the `&[class]` block (~9 lines) from `EditionRailCard.recipe.ts` entirely; the same values move to live where the rest of the ladder already lives. No visual change — same computed CSS, no more cascade-order dependency to worry about.

### ADV-2 — `& h2` / `& .badge` reach past `EditionTheme`/`Badge`'s recipe boundary into their rendered internals
**file:line**: `EditionRailCard.recipe.ts:51-56` (`upcoming.root`)
**axis**: adversarial
**risk**: no-op (fix preserves current visuals, changes only how they're reached)
**smell**: Two selectors style a sibling component's internals by guessing at its rendered tag/class rather than going through its prop/variant API — one of them works only by accident.

**evidence**: `'& h2': { color: 'muted' }` (`:51`) depends on `EditionTheme` rendering its heading as a literal `<h2>` — true today because `EditionRailCard.tsx:64` doesn't pass `as`, so `EditionTheme`'s default (`as: Tag = 'h2'`, `EditionTheme.tsx:48`) applies. `EditionTheme`'s own `as` prop is typed `'h1' | 'h2'` specifically because different call sites need different heading levels (`Hero.tsx:38` passes `as="h1"`) — if an accessibility pass ever changes the rail's heading level, this selector silently stops matching with no type error and no visual regression signal. `'& .badge'` (`:52-56`) reaches into `Badge`'s rendered className (`className: 'badge'`, `src/design-system/recipes/badge.ts:10`) to override `background`/`borderColor`/`color` from outside, rather than through `Badge`'s own `tone` variant (`highlight` | `outline`, `badge.ts:32-38`) — the mechanism this exact styling need already exists for.

**recommendation**: For `& .badge`: add a third `tone: 'muted'` variant to `src/design-system/recipes/badge.ts` (bg `gray.900` / border `gray.700` / color `gray.400`, matching this override exactly) and have `EditionRailCard.tsx` pass `tone={status === 'upcoming' ? 'muted' : ...}` down through wherever the year/status `Badge`s render (`EditionRailCard.tsx:70-75`), deleting the `& .badge` block. For `& h2`: since `EditionTheme` has no existing prop for "mute the whole heading text," either add one (a boolean `muted` prop, orthogonal to `accent`) or, more simply, target `styles.heading` (the class `EditionTheme` already exposes as an addressable slot, not the `h2` tag) — still not ideal cross-recipe reach, but removes the tag-coupling fragility; the variant-prop route is the fuller fix. No visual change either way.

### ADV-3 — reveal transition duplicates the existing `enter` animation model, and hardcodes a value the DS already named
**file:line**: `EditionsNav.recipe.ts:13-19`
**axis**: adversarial
**risk**: no-op for the literal-only fix; visual-risk if fully composed onto `animationStyle: 'enter'` (see below)
**smell**: A hand-rolled "fade + translateY, staggered by `--i`" transition re-implements a system that already exists as a named `animationStyle`, and its per-card stagger literal (`60ms`) is the exact value of an existing duration token, used unreferenced.

**evidence**: `transition: '[opacity {durations.reveal} {easings.expo} calc(var(--i, 0) * 60ms), transform {durations.reveal} {easings.expo} calc(var(--i, 0) * 60ms)]'` (`:16-17`) hardcodes `60ms` where `tokens.durations.stagger` (`tokens.ts:199`) is already `60ms` — and the formula `calc(var(--i, 0) * {durations.stagger})` already exists verbatim as `animationStyles.enter.DEFAULT.animationDelay` (`tokens.ts:361`). The rest of the mechanism — opacity 0→1, a translate offset resting state, a `_motionReduce` override killing the effect — is also what `animationStyle: 'enter'` already provides via the `enter` keyframe (`tokens.ts:58-66`) and its baked-in `_motionReduce: { animation: 'none' }` (`tokens.ts:363`), parameterized by the `--enter-y` CSS var precisely so different call sites can use different offsets without re-declaring the mechanism (this file uses `16px`; `enter.DEFAULT` defaults to `30px`).

**recommendation**: Minimal fix (no visual change): replace the bare `60ms` with `{durations.stagger}` in the `calc()`. Fuller fix (shrinks more, but changes the animation model): replace the whole custom `opacity`/`transform`/`transition`/`_motionReduce` block with `animationStyle: 'enter'` plus `'--enter-y': '16px'` under the `[data-revealed=true] &` selector, deleting the hand-rolled twin entirely (~6 lines) and its separate `_motionReduce` override (already handled by the shared style). Flag: `enter.DEFAULT` uses `durations.entrance` (900ms) where this file currently uses `durations.reveal` (600ms) — composing onto the shared style as-is would slow the reveal down, so the full fix needs either accepting that timing change or teaching `animationStyle: 'enter'` to take a duration override too. Not a drop-in swap; call out before applying.

**risk**: no-op

**recommendation**: Delete the line. If this is also getting folded into a new `tone: 'muted'` `Badge` variant per ADV-2's recommendation, simply don't carry `boxShadow` into that variant either. No visual change — the line does nothing currently.
