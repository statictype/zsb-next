# Component audit — `src/components/ui/` + `src/components/Manifesto/` (refactor-first)

Scope: `Accordion`, `Badge`, `Button`, `Card`, `Checkbox`, `Collapsible`, `Dialog`,
`Eyebrow`, `LinkList`, `SectionHeading` (all under `src/components/ui/`) and
`Manifesto`. Grounded against `src/design-system/tokens.ts`, `preset.ts`,
`panda.config.ts` (`strictTokens: true`, "every style value is a token or a
deliberate `[bracketed]` exception — no load-bearing exception that stays raw"),
and every call site of each component under `src/`.

`LinkList` produced no findings that survived verification — its recipe's
bracket usages (`transitionProperty: '[padding-left]'`, `'[transform]'`, …) are
necessary escapes proven so by sibling `transitionProperty: '[...]'` usage
across a dozen other recipes (Panda's `transitionProperty` utility only
recognizes named shortcuts like `colors`/`common`, not arbitrary single
properties) — it stays out of the table below.

### DS-1 — `src/components/Manifesto/Manifesto.recipe.ts:33`
**axis:** ds-hygiene
**smell:** `lineHeight: { base: 'tight', md: 'display', '4xl': '[1.08]' }` — the `4xl` step is a raw bracketed literal.

**evidence:** No other `1.08` line-height exists anywhere in `src/` (grepped). The
existing `lineHeights` scale has `tight: 1.16` as the nearest member (~7%
delta) — at this slot's `4xl` fontSize (`clamp(48px, …, 74px)` per the
`title` slot's own `fontSize` ramp) that delta is visible but not dramatic.
Per the no-exceptions policy, priority order puts "snap to nearest token,
accept a small visual delta" *above* minting a new single-use token.

**recommendation:** Replace `'[1.08]'` with `'tight'` (deletes the last raw
literal from this slot). This is a value change (1.08 → 1.16, ~7% looser
line-height at the largest breakpoint) — worth an eyeball before committing.
If the visual delta is rejected, the fallback is a new `lineHeights.tighter:
'1.08'` token (still no bracket, still discoverable) — not restoring the
bracket.

**smell:** `40px` is independently reinvented in 3 places with no token backing it.

**evidence:**

call site for the same token in a follow-up (out of this review's scope but

### STRUCT-1 — `Accordion.tsx:54-56` + `Collapsible.tsx:41-43` + `accordion.ts:47-54` + `collapsible.ts:43-50`
**axis:** structure
**smell:** Two components independently ship the same disclosure-chevron shape.

**evidence:** `accordion.ts`'s `itemIndicator` slot:
```
{ display: 'inline-flex', flexShrink: '0', color: 'muted',
  transition: 'transform {durations.fast} {easings.quint}',
  '&[data-state=open]': { transform: 'rotate(180deg)' },
  _motionReduce: { transition: 'none' } }
```
`collapsible.ts`'s `indicator` slot:
```
{ display: 'inline-flex', marginLeft: 'auto', color: 'inherit',
  transition: 'transform {durations.normal} {easings.quint}',
  '&[data-state=open]': { transform: 'rotate(180deg)' },
  _motionReduce: { transition: 'none' } }
```
Same shape (flex + rotate-180-on-open + motion-reduce kill), differing only in
`color`/`duration`/`marginLeft`. The JSX wrapping is duplicated verbatim too:
`<Ark*.Indicator className={styles.indicator}><RiArrowDownSLine size={20}
aria-hidden /></Ark*.Indicator>` appears at `Accordion.tsx:54-56` and
`Collapsible.tsx:41-43` with no shared source.

**recommendation:** Extract the shared rotate/motion-reduce mechanics into one
`layerStyles.disclosureIndicator` (display, transition-property, the
`[data-state=open]` rotate, `_motionReduce`), consumed by both `accordion.ts`
and `collapsible.ts`, leaving only `color`/`duration`/`marginLeft` as
call-site deltas. Collapses ~10 duplicated lines across 2 recipe files to one
shared definition + two thin overrides. (The duplicated `RiArrowDownSLine
size={20}` JSX is small enough that a shared icon constant is optional, but
worth doing in the same pass since it's the same extraction boundary.)

### ADV-1 — `Badge.tsx:9`, `Button.tsx:6-8`, `Eyebrow.tsx:9`
**axis:** adversarial
**smell:** All three "ZSB-71 unified primitive" docstrings describe a richer variant API than what their recipes actually ship — dead documentation, not settled rationale.

**evidence:**
- `Badge.tsx:9` — *"Visual variants live in the Panda `badge` recipe (`tone` ×
  `size` × `elevated`)."* `badge.ts` defines exactly one variant: `tone`. No
  `size`, no `elevated`.
- `Button.tsx:6-8` — *"Button — the one action primitive (ADR 0019): primary |
  secondary | ghost | text | icon × size."* `button.ts`'s actual `variant`
  union is `primary | secondary | link | icon` — no `ghost`, no `text`.
- `Eyebrow.tsx:9` — *"Variants (`tone` × `size` × `rule`) live in the Panda
  `eyebrow` recipe."* `eyebrow.ts` defines exactly one variant: `rule`. No
  `tone`, no `size`.
- The `ghost`/`text` naming isn't confined to the docstring: a consumer,
  `EventModal.tsx:15-17`, comments *"The Back/Share controls are plain ghost
  `<Button>`s…"* for controls that are actually `variant="secondary"` at
  `EventModal.tsx:44,48`. The stale name propagated past the component
  boundary.

**recommendation:** Correct (or delete) the variant lists in all three
docstrings to match the shipped recipe shape — `tone` only for Badge, `rule`
only for Eyebrow, `primary | secondary | link | icon` for Button. Per the
audit's one rule, a comment describing an API that doesn't exist isn't a
documentation nice-to-have to leave for later, it's actively wrong and should
be fixed in the same pass as any other finding here.

### PROP-2 — `src/components/ui/Card/Card.tsx:14-23` / `src/design-system/recipes/card.ts`
**axis:** prop-api
**smell:** `Card`'s polymorphism (`as`/`href`) doesn't type per-element attributes, so a caller needing anchor-only props (`target`, `rel`) can't use the component at all.

**evidence:** `CardProps` is `CardVariantProps & HTMLAttributes<HTMLElement> &
{ as?: ElementType; href?: string }` — `HTMLAttributes<HTMLElement>` has no
`target`/`rel`, and `as` doesn't narrow the prop type to the element it's
set to. `ExternalGallery.tsx:29-36` hits this directly and says so in its own
comment: *"The unified hairline Card (ZSB-71) via the recipe directly — a raw
external `<a>` (target/rel) that the `<Card>` wrapper's props don't cover."*
It then imports the `card` recipe function straight from `styled-system/recipes`
and hand-rolls a second, untyped rendering path (`<a className={cx(card({
ground: 'onDark', interactive: true }), styles.card)}>`) for the exact
primitive `Card` exists to unify. `Button` already solved this identical
problem: `Button.tsx`'s `asChild` prop (lines 22-26, 39-41) merges the
recipe's className onto a caller-owned child element via `cloneElement`,
so element-specific props (including `target`/`rel`) stay on the child's own
native type.

**recommendation:** Add the same `asChild` escape hatch to `Card`, mirroring
`Button`'s implementation, instead of the weaker `as`+manual-prop-list
pattern. This lets `ExternalGallery.tsx` delete its entire hand-rolled block
in favor of `<Card asChild ground="onDark" interactive><a href={href}
target="_blank" rel="noopener noreferrer">…</a></Card>` — removes ~7 lines
at that call site and the second, drifting code path entirely. No rendered-
output change: the resolved class list is identical either way.

### DS-4 — `src/design-system/recipes/badge.ts:16-30`
**axis:** ds-hygiene
**smell:** The base block has 6 raw literals; 2 have exact matches in the existing token scale that go unused.

**evidence:**
- `fontWeight: '600'` — exact match to `fontWeights.semibold` (`'600'`).
- `letterSpacing: '1.2px'` — exact match to `letterSpacings.label` (`'1.2px'`).
- `borderWidth: '0.5px'` — exact numeric match to
  `borderWidths.gradientRingHairline` (`'0.5px'`), but that token's name
  scopes it to the hover gradient-ring mechanism (its only other consumer is
  `Calendar.recipe.ts:160`) — reusing it as-is under Badge would be a
  semantic-token misuse in the other direction (a "ring" token on a static
  chip border).
- `fontSize: '10px'`, `paddingInline: '12px'`, `paddingBlock: '6px'`,
  `lineHeight: '1.3'` — no scale member is close enough to snap to (nearest
  `lineHeights` member is `tight` at 1.16; nearest `fontSizes` reference is
  the *stepped* `xs` (`{base:'10px', lg:'11px'}`), which Badge deliberately
  doesn't use since it's documented as one fixed size, not a responsive one).

**recommendation:**
- Swap the two exact matches immediately: `fontWeight: 'semibold'`,
  `letterSpacing: 'label'`.
- Generalize `borderWidths.gradientRingHairline` to a neutral name (e.g.
  `borderWidths.hairlineThin`) and point both `badge.ts` and
  `Calendar.recipe.ts:160` at the renamed token — net token count unchanged,
  removes the misleading name at both call sites.
- Add new single-use tokens for `fontSize`, `paddingInline`, `paddingBlock`,
  `lineHeight` (Badge is a hand-tuned, non-ramped primitive with no sibling
  literal to merge against — per the policy, a named single-use token still
  beats an inline bracket).

### PROP-3 — `src/design-system/recipes/badge.ts` (evidence at `EditionRailCard.tsx:73-83`, `EditionRailCard.recipe.ts:29-33`)
**axis:** prop-api
**smell:** `badge.ts`'s `tone` variant only offers `highlight`/`outline`; a consumer needing a third "muted" look reaches for a `className` override instead of a recipe variant.

**evidence:** `EditionRailCard.recipe.ts:29-33` defines its own `badgeMuted`
slot (`background: 'gray.900', borderColor: 'gray.700', color: 'gray.400'`),
applied via `className` to two separate `<Badge>` calls at
`EditionRailCard.tsx:73` and `:77-80` whenever `status === 'upcoming'` — the
exact "ad-hoc `className` override that should be a recipe variant" pattern
this axis targets. The recipe's own comment even frames it as deliberate:
*"an opt-in override through Badge's own prop, not a `.badge` selector
guessing at its rendered class from outside"* — i.e. it already knows this
belongs on `Badge`, not beside it.

**recommendation:** Add `tone: 'muted'` to `badge.ts` (values lifted verbatim
from `badgeMuted`), then delete the `badgeMuted` slot from
`EditionRailCard.recipe.ts` and both conditional `className` expressions at
the call site — `className={isUpcoming ? styles.badgeMuted : undefined}` on
two `<Badge>`s collapses to `tone={isUpcoming ? 'muted' : undefined}` (or
folds into the existing `tone={isUpcoming ? undefined : 'outline'}`
ternary on the second one). Net: one new named tone on the shared primitive,
minus a local slot and two conditional expressions at the call site.

### DS-12 — `src/design-system/recipes/dialog.ts:25`
**axis:** ds-hygiene
**smell:** Same `outline` duplication as DS-3, plus a negative `outlineOffset` that's a derivable function of an existing token, restated as an independent literal instead.

**evidence:** `content._focusVisible: { outline: '2px solid
token(colors.highlight)', outlineOffset: '-2px' }`. The `-2px` is exactly the
negation of `borderWidths.focus` (`2px`) — this dialog content sits flush to
its own edge, so the ring needs to draw inward rather than outward, unlike
every other `_focusVisible` site in this review. That's a legitimate
call-site delta in *direction*, but the *magnitude* (`2px`) shouldn't be a
second independent literal that can silently desync from `borderWidths.focus`
if that token's value ever changes.

**recommendation:** `outline: 'focus'` (same fix as DS-3/8/11) for the
shorthand; for the offset, derive it instead of restating it:
`outlineOffset: 'calc(token(borderWidths.focus) * -1)'`. Zero visual diff
today (`borderWidths.focus` is exactly `2px`), but the two values can no
longer drift apart.

### DS-13 — `src/design-system/recipes/dialog.ts:27-37`
**axis:** ds-hygiene
**smell:** The `title` slot hand-rolls the classic visually-hidden ("sr-only") CSS recipe entirely from raw literals, with no token or `layerStyle` backing any of it.

**evidence:**
```
title: {
  position: 'absolute', width: '1px', height: '1px', padding: 0,
  margin: '-1px', overflow: 'hidden', clip: 'rect(0 0 0 0)',
  whiteSpace: 'nowrap', borderWidth: 0,
}
```
This is a well-known, standard a11y pattern (not dialog-specific), currently
used exactly once in `src/` — but it's exactly the kind of small, easily
reused utility the DS's `layerStyles` category exists for (see
`layerStyles.skeleton`/`chipRow`/`brushStrokeRule` for precedent: single-
mechanism, cross-cutting visual/behavioral recipes named once).

**recommendation:** Extract as `layerStyles.srOnly` even though it's
currently single-use — per the policy, a named token/layerStyle beats an
inline trick nobody will find the second time they need to hide something
visually while keeping it in the accessibility tree.
